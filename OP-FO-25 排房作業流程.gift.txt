$CATEGORY: OP-FO-25 排房作業流程

::OP-FO-25-Q01::在執行排房作業（流程 1）時，若遇到同一姓名有多筆訂單，應優先採取的排房原則為何？ {
~將其安排在不同樓層，分散音量影響#SOP 明確指出應儘量安排在同一樓層以方便房客，分散樓層會增加房客往返的困擾。
=優先排房，並儘量安排在同一樓層#根據排房流程第 3 點，多筆同名訂單應先行排房，並以同樓層為原則，方便顧客並利於管理。
~待所有單筆訂單排完後，再處理多筆訂單#作業規範要求同名多筆訂單者應「先行排房」，而非最後處理，以確保房源充足且能連號或同層。
~隨機分配房號，之後再由早班統一調整#流程強調需確認資訊正確即完成，且多筆訂單有特定排序邏輯，並非隨機分配後再調整。
}

::OP-FO-25-Q02::根據排房作業注意事項，若房況許可，應儘量「避免」安排房客入住下列哪些房號？ {
~位於電梯正對面的房間#資料中未提到避開電梯對面房間，而是強調避開逃生設備房與角落房。
~高樓層的景觀房#高樓層通常是特殊需求（如：高樓層需求）的安排對象，並非需避開的房型。
=避難逃生設備房間（如 202、305）與角間#注意事項第 3 點明確列出應避開 202、305、606、807 等避難設備房及角間，以維護住客體驗或安全動線。
~靠近員工休息區或備品間的房間#資料中並未規範需避開備品間鄰近區域，其重點在於設備房與建築角落。
}

::OP-FO-25-Q03::列印 R 卡（旅客登記卡）時，若遇到同批顧客有多間房間，應如何在系統中操作以避免重複或浪費？ {
~勾選「全選」後直接列印，系統會自動整合#系統不會自動整合，直接列印會導致每間房都印出一張含有重複資訊的登記卡。
=點選全選後，將該批顧客中「數字最大」的房號取消勾選#流程 2 第 6 點指出，應取消勾選數字最大的房號，此為避免同名多筆訂單重複列印的標準步驟。
~點選全選後，手動刪除多餘的房號資訊#SOP 要求在列印前的清單中取消勾選特定房號，而非列印後才手動刪除。
~每一間房號都必須單獨分開列印，不可使用全選#注意事項提到多筆同名訂單只選一個列印，以避免浪費，而非全部單獨列印。
}

::OP-FO-25-Q04::當訂單來源顯示為 Agoda、Trip、Klook 或 Expedia 串單至系統時，在製作登記卡時應如何標註金額資訊？ {
~註記「已付」並寫上訂單上的實際房價#補充說明 1 明確規定針對這些平台串單，備註需寫「已付」但「不可以寫出金額」。
=註記「已付」且「不可以」寫出金額#這是針對 OTA 第三方代收訂單的嚴格規定，避免因平台價差引起顧客爭議。
~註記「未付」，待客人入住時現場收費#Agoda 等串單通常已在平台付款，應註記為「已付」以符合帳務狀態。
~不需做任何備註，直接空白即可#為了讓櫃檯人員一眼辨識帳務狀態，必須寫上「已付」等狀態字樣。
}

::OP-FO-25-Q05::在填寫房況表時，若某一房號為「當日退房」，應使用哪種顏色的螢光筆進行標示？ {
~紅色螢光筆#紅色螢光筆是專門用於標示「留房」的房號，而非退房。
~黃色螢光筆#流程中並未提到黃色螢光筆的使用，標準作業應使用綠色或紅色。
=綠色螢光筆#流程 5 第 5 點明確規定：當日退房的房號上畫綠色螢光筆。
~藍色螢光筆#藍色螢光筆並非房況表標準化標記顏色，流程指定為綠色。
}

::OP-FO-25-Q06::關於 9.602 房型的特殊作業需求，在房況表上應該確認哪些項目？ {
~確認是否需要加床#資料指出 602 房型需確認的是 SF 或 DD，而非一般加床服務。
=確認當日需做 SF 或 DD，並填寫於表格#流程 5 第 9 點提到 602 為特殊房型，需確認 SF 或 DD 並在查房狀況表上填寫。
~確認是否為外國人使用並註記星號#註記星號是針對「所有」外國人使用的退房，而非 602 房型特有的確認項目。
~確認是否需要更換為併床模式#併床需求是針對 CT 房型（兩小床）的作業，而非 602 房型的特殊要求。
}

::OP-FO-25-Q07::在德安飯店前台系統中，若要確認某筆訂單是否已經完成「排房」，應查看預進列表中的哪個數字？ {
=若排房數字為 1 代表已排房，0 代表未排房#根據流程 1 第 5 點，系統中排房數字 1 代表已完成排房，0 則代表尚未分配房號。
~若排房數字為 1 代表已入住，0 代表預進#此數字反映的是排房狀態，而非顧客是否已經辦理 C/I 入住。
~數字代表該訂單預計入住的總天數#系統中的排房狀態欄位是用來檢核是否已指派房號，而非天數。
~數字代表該筆訂單所含的總房間數#流程明確定義此 0/1 狀態為「已排房」與「未排房」的判斷依據。
}

::OP-FO-25-Q08::若房客在官網訂購「高鐵聯票」，根據作業指導書的備註金額填寫規範，應如何處理？ {
~寫出房費加上高鐵票價的總金額#補充說明 2 提到不可寫出房價加高鐵的總金額，以免產生退票或帳務爭議。
=僅需寫出總房價，勿寫出高鐵金額#補充說明 2 規定備註金額只需寫出總房價，不應包含高鐵費用（註：雖 2026/5/2 後已無此專案，但作業標準仍以此為準）。
~僅寫出高鐵票價，房價部分註記已付#規範要求標註房價而非票價，以便櫃檯核對住宿部分的款項。
~比照 Agoda 串單模式，完全不寫金額#官網訂單有不同的顯示規則，應寫出總房價以利核對，與 OTA 的隱藏金額做法不同。
}

::OP-FO-25-Q09::在填寫房況表（流程 5）時，關於「備品」的備註作業，下列敘述何者錯誤？ {
~入住房客需要的備品名及數量需寫在備註格中#這符合流程 5 第 3 點的正確操作，應寫上如「床圍*1」等資訊。
=退房時須收取的備品（如澡盆）不需記錄在房況表上#此敘述錯誤。流程 5 第 4 點規定：退房須收取的備品名及數量也必須寫在備註格中。
~若有留房（續住）的情況，也必須寫上須收備品#這是正確的操作規範，確保在續住期間對借出備品進行控管。
~外國人使用之退房房間須額外註記星號#這是流程 5 的特殊要求，用以提醒房務或櫃檯注意特定國籍服務差異。
}

::OP-FO-25-Q10::在製作旅客住宿登記卡時，如何判別應在備註欄寫上「已付」、「未付」或「已授權」？ {
~根據顧客的國籍與性別判斷#付款狀態與客人的個人特徵無關，而是取決於訂單帳務資訊。
~根據訂房來源是否為官網判斷#官網訂單也可能有已付或現場付的情況，需確認具體帳務欄位。
=確認應收訂金欄的已付金額與房費對比，以及備註有無預先授權#流程 3 第 6-7 點指出，應比對已付金額與右邊房費金額，並檢查備註是否有 Pre-auth 紀錄來判定。
~由德安系統自動在 R 卡上產生，不需人工判別#SOP 明確要求夥伴應依照情況判別後，在登記卡備註上「寫上」對應狀態。
}
