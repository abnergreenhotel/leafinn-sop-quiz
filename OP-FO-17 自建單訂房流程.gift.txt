$CATEGORY: OP-FO-17 自建單訂房流程

::OP-FO-17-Q01::當您在接聽訂房電話時，若因故無法立即登入德安系統輸入資料，根據標準流程應如何處理？ {
~請客人稍後再撥，待系統準備好後再服務#這會導致客人的不便並可能流失訂單，非手冊建議的做法。
=先拿出紙本訂房單，一邊聽取需求一邊填寫#手冊明確指出若無法同時輸入德安，可先以訂房單記錄客人需求。
~直接憑記憶背下所有需求，掛電話後再補登#人類記憶有限，容易漏掉日期或房型等重要細節，增加出錯風險。
~請客人改發送 FB 或 LINE 訊息以利後續作業#當客人已經致電時，應優先在電話中提供即時服務而非推託。
~要求另一位同事放下手邊工作協助登入#這會干擾其他同事的作業流程，且手冊已有個人因應措施。
}

::OP-FO-17-Q02::在進行「新增訂房」時，若需查詢當下館內可賣出的房間數量與網路平台剩餘數，應點選哪一個功能？ {
~房價代號查詢#房價代號用於選擇報價專案，而非查詢剩餘庫存數。
~客房維修管理#此功能與房況維修有關，非訂房時的庫存查詢步驟。
=鎖控分頁#手冊說明可按下新增訂房後，點選下方分頁「鎖控」來查看庫存與 WRS 剩餘數。
~歷史旅客查詢#這用於找尋曾入住客人的 Profile 資料，非房量庫存資訊。
~簡訊發送紀錄#這是用於確認匯款資訊發送狀況，與查房量無關。
}

::OP-FO-17-Q03::關於庫存數計算，若畫面上顯示「庫存數」為 2，「WRS 鎖控剩餘數」為 3，則目前總計尚未售出的房間數是多少？ {
~2 間#此數值僅代表館內數量，未包含網路平台的數量。
~3 間#此數值僅代表網路可售數量，未包含館內實體剩餘庫存。
=5 間#計算公式為庫存數加上 WRS 鎖控剩餘數等於總計尚未售出房間數。
~1 間#這是相減的錯誤計算，不符合手冊的加總邏輯。
~0 間（代表已爆房）#只有在庫存數下方小計為負數時才可能代表爆房。
}

::OP-FO-17-Q04::手冊強調「庫存數」最下方的小計處絕對不能出現什麼情況，否則表示可能發生爆房？ {
~數值等於 0#數值為 0 是應保持的標準，代表剛好售完。
~數值大於 10#這僅代表目前空房很多，並非異常狀況。
=數值為負數#手冊註記小計處若為負數，表示超賣或爆房情況發生。
~數值與 WRS 相等#這只是庫存分布的巧合，與爆房判斷無關。
~數值變為紅色#手冊主要以正負數值作為判斷準據，非顏色提示。
}

::OP-FO-17-Q05::若顧客詢問公司簽約特約價，但在名單中不確定其公司名稱，應如何在系統中輔助查詢？ {
~在訂房公司格內直接輸入完整的公司負責人姓名#手冊並無提到以負責人姓名查詢特約名單的步驟。
=在訂房公司格內輸入「%」，再輸入公司統編#手冊明確指導可以使用 % 符號搭配統編來查詢特約商。
~撥打總部電話請求查詢該公司合約狀態#櫃檯應先自行在系統查詢，若無資料才由總部處理，而非第一步就撥打。
~要求客人將合約書正本拍照上傳至 LINE#這會增加客人的作業麻煩，應優先使用系統查詢統編。
~告知客人必須提供實體名片才能查詢價格#電話訂房時無法即時核對名片，系統查詢統編才是標準做法。
}

::OP-FO-17-Q06::在自建訂單流程中，手動保留一間標準雙人房後，應去哪裡調整網路販售數量以避免重複預訂？ {
~德安前檯系統的客務帳務分頁#這是處理結帳的系統，不負責網路庫存調控。
=進入「網路訂房系統」中的「房種庫存管理」#手冊規定需回到主選單進入網路訂房系統，手動扣除網路庫存以成功保留。
~直接在現場排房畫面中鎖定房間#現場排房無法直接連動扣除 OTA 網路平台的剩餘房量。
~撥打網路平台業務電話要求下架房間#此操作太耗時，應由內部網路訂房管理系統自行操作。
~調整「鎖控」分頁中的所有庫存為 0#這樣會導致全館停止對外銷售，不符合單一訂單的操作邏輯。
}

::OP-FO-17-Q07::若網路房種庫存管理顯示為「1」，更改成「0」存檔後，卻又自動跳回「1」，這代表什麼意義？ {
~系統故障，需重啟德安伺服器#這通常是業務邏輯狀況，非系統硬體故障。
=該房間剛好被網路客人訂走，只是尚未串單#手冊提醒若數字無法更改為 0，代表該房已被搶先預訂，訂單串接中。
~該房型已被列為禁賣房，無法手動更改#手冊並無提到禁賣房會導致數字跳回的設定。
~櫃檯操作權限不足，無法變更數字#若是權限不足應是無法存檔，而非存檔後數值跳回。
~代表該房型為保證配額，不能清空#手冊解釋此狀況是因為網路訂單已經進入流程，並非配額限制。
}

::OP-FO-17-Q08::在 Full Name 欄位輸入「%旅客姓名」按下 Enter 的主要目的是什麼？ {
~為了讓系統自動生成隨機訂單編號#訂單編號由系統自動產生，與此輸入動作無關。
=系統會自動尋找是否以前有該名旅客的資料#手冊註記在姓名前加 % 可讓系統自動比對歷史旅客 Profile。
~將該訂單標註為需要主管特別簽核#「%」符號在德安系統中多為查詢前綴，而非標記權限。
~強制系統跳過電話號碼的必填檢查#Profile 建立仍需輸入必要聯繫資訊，此符號僅供搜尋。
~將姓名自動翻譯成英文拼音#系統無自動翻譯功能，需人工輸入姓名。
}

::OP-FO-17-Q09::若顧客需求「標準雙人房、高樓層、床圍*2」，應如何在系統中記錄這些租借備品需求？ {
~僅寫在櫃檯交接本上即可#資訊必須電子化記錄在訂房單中，才能確保所有人員都能看見。
=在備註欄輸入備品後，點選「櫃檯備品」扣除使用數#手冊步驟 9 指導需先在排房備註輸入備品名，再到櫃檯備品功能扣除數量。
~將備品需求直接寫在旅客的姓名後面#這會導致訂單姓名錯誤，且無法追蹤備品庫存。
~要求客人在入住前三小時再致電告知備品需求#為提供良好服務，應在訂房時即完成記錄與保留。
~在房價欄位中加上備品的租借費用#除非該備品需額外收費，否則不應隨意更改房價欄位。
}

::OP-FO-17-Q10::當客人表示開車前來但還不確定車號時，正確的系統備註與溝通方式為何？ {
~不進行任何備註，等客人抵達現場再說#這會導致無法預留停車格或預估車位狀況。
=在備註欄打上「會開車」，並告知當天入住時再確認車號#手冊說明若車號不確定，應先備註會開車，當天再確認即可。
~強制要求客人必須在當天下午五點前回報車號#這會給客人壓力，非手冊建議的柔軟話術。
~先隨便輸入一個錯誤車號，存檔後再修改#這會導致資料錯誤，不符合誠實記錄原則。
~告知客人若無車號將無法提供特約停車位#飯店應提供預留備註，而非因無車號就直接拒絕服務。
}

::OP-FO-17-Q11::針對「假日」入住的客人，手冊特別建議可以提醒客人什麼細節，以避免停車位客訴？ {
~提醒客人飯店對面有收費停車場可以自費停放#這無法解決特約車位滿位的不滿，應建議如何避開尖峰。
=提醒客人避開用餐時段過來停車，較容易有位置#手冊注意事項提到假日可建議避開用餐時段，以減少客訴。
~提醒客人若無車位需自行在路邊尋找車位#這類負面提醒容易引起反感，應優先提供積極建議。
~建議客人改搭大眾運輸工具過來飯店#對於已經打算開車的客人，此建議不切實際且缺乏彈性。
~告知假日停車需額外支付每小時 100 元的費用#手冊未提及此收費規定，應專注於停車時段的建議。
}

::OP-FO-17-Q12::在操作信用卡預先授權時，卡機應選擇哪一個功能選項？ {
~1. 一般結帳#這是實際刷卡交易，並非預先授權。
~2. 退貨/取消#這是退款功能，非建立授權。
=5. 預先授權#手冊明確註明卡機應選擇「5.預先授權」進行保證訂房作業。
~9. 結帳清帳#這是日結或批次上傳功能，與訂單授權無關。
~3. 離線授權#離線授權用於特殊狀況，標準流程應選預先授權。
}

::OP-FO-17-Q13::若選擇銀行匯款作為保證訂房方式，應提醒客人在幾日內完成匯款？ {
~入住當天下午三點前#這對飯店而言太遲，無法發揮保證訂房的作用。
~收到簡訊後的 24 小時內#這時限過於急迫，手冊有較寬裕的期限規定。
=今日後三日內#手冊明確記載提醒客人需於三日內完成匯款並回電告知後五碼。
~入住日前一週完成即可#若訂單是近期產生的，此期限不具體。
~訂單成立後的兩週內#這對庫存控管來說過長，非標準期限。
}

::OP-FO-17-Q14::完成銀行匯款簡訊發送後，通話中應要求客人匯款後執行什麼動作？ {
~回傳匯款收據的照片到 IG#IG 非正式查帳管道，且手冊要求的是回電告知後五碼。
=來電告知匯款帳號後五碼#手冊指示需請客人回電告知後五碼以便查帳入帳。
~不需做任何動作，飯店系統會自動對帳#飯店需要後五碼進行人工對帳，非完全自動化。
~在匯款單備註姓名後傳真過來#傳真已較少使用，回電告知後五碼較為快速便利。
~將實體存摺拍照上傳至 FB 私訊#這涉及過多隱私且非標準作業流程。
}

::OP-FO-17-Q16::在電話結束前，針對「取消或更改訂房」的期限，標準話術應如何提醒？ {

~一旦保證訂房成功，即不可取消或退費#這是不公平條款，手冊有提供合理的取消期限。
~前七天內取消需扣除全額房費#手冊並未提到如此嚴格的扣款規則。
~僅限更改一次日期，不可取消#手冊允許在期限內免費取消。

::OP-FO-17-Q18::若旅行社（車隊）來電訂房，櫃檯第一步應如何確認身分以給予特約價？ {

::OP-FO-17-Q19::當車隊司機來電訂房且公司已簽約，辦理入住時櫃檯應如何處理證件？ {
