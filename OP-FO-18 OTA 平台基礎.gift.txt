$CATEGORY: OP-FO-18 OTA 平台基礎

::OP-FO-18-Q01::當 Booking.com 訂單的信用卡授權失敗時，根據標準作業流程，下列哪一項處置方式最正確？ {
~直接在飯店前檯系統取消該筆訂單以釋出房源#隨意取消訂單可能導致客訴與平台規範問題，應先依程序通知顧客更換卡片。
~等待顧客入住時再現場支付，不需預先處理#流程規定需確認匯款或預取授權是否成功，若失敗應主動聯繫以降低爽約風險。
=至 Booking.com 後台留言告知更換信用卡，並標註信用卡無效#這是針對 Booking.com 授權失敗時的標準報備與通知程序。
~僅撥打電話聯繫，若未接聽則視為自動放棄訂房#除電話聯繫外，流程亦要求在後台留言，不能單方面視為放棄。
~私下聯繫顧客提供匯款帳號，請顧客改為轉帳支付#流程建議先要求更換信用卡，並非直接跳轉至非平台的私人轉帳。
}

::OP-FO-18-Q02::若 Booking.com 訂單中未提供顧客電話資訊，房務人員應如何進行聯繫？ {
=在 Booking.com 後台留言聯繫顧客#流程規定當無電話資訊時，應利用平台後台的留言功能進行溝通。
~直接聯繫平台客服，要求提供顧客的私人聯絡電話#流程首選是先在後台留言，而非第一時間要求客服提供私人個資。
~暫緩處理，等顧客主動聯繫飯店確認行程#飯店需在入住前主動確認資訊，被動等待會影響作業排程與顧客體驗。
~透過電子郵件發送確認信件，不再使用平台系統#流程強調使用平台後台留言功能作為無電話時的主要替代方案。
~若無電話則無法聯繫，直接將該訂單標記為待定#無電話並不代表放棄聯繫，仍有後台留言管道可供使用。
}

::OP-FO-18-Q03::在與 AGODA 或 EXPEDIA 的顧客進行訂房確認時，下列哪一項是必須特別核對的重點？ {
~確認顧客是否需要預約機場接送服務#雖然接送是加值服務，但非流程中針對這兩個平台強調的必核項目。
=與顧客確認訂單內容是否包含早餐#針對 AGODA、EXPEDIA 與 Booking.com 訂單，確認含早與否是流程指定的必要動作。
~核對顧客在該平台的會員等級以提供升等#流程聚焦於訂單基礎資訊核對，而非會員權益或房型升等。
~詢問顧客是否為第一次入住綠宿行旅#這屬於一般關懷，非流程手冊中明確要求的作業核對項目。
~要求顧客提供護照或身分證字號以先行建檔#入住確認通常不需在電話中收集敏感個資，流程亦未對此規範。
}

::OP-FO-18-Q04::針對官網訂單的確認，流程中特別強調需要與顧客核對哪一項資訊？ {
=確認顧客的訂購專案內容#官網訂單通常包含多樣化的活動或專案，流程規範需確認其專案內容。
~詢問顧客是透過哪一種搜尋引擎找到官網#此為行銷調查數據，非前檯訂房確認作業的標準核心項目。
~核對官網折扣碼是否輸入正確#折扣核對應在訂單成立時已完成，確認流程主要在於專案內容的溝通。
~確認顧客是否有加入官網 LINE 好友#此非標準確認流程中要求的必要核對事項。
~詢問顧客是否願意將早餐更換為下午茶#流程並未提到變更餐飲類型的選項，而是應確認原訂單專案內容。
}

::OP-FO-18-Q05::在「入住前七日」聯繫顧客時，除了確認訂單資訊與特殊需求外，下列哪兩項資訊是主要詢問重點？ {
=預抵時間及停車需求#流程明確規定入住前七日需詢問顧客預計抵達的時間以及是否有停車位需求。
~用餐偏好及過敏食物#雖然重要，但流程手冊將重心放在抵達時間與硬體設施（停車）的需求。
~發票抬頭及統編資訊#發票資訊通常於退房或辦理入住時確認，非七日前聯繫的主要項目。
~是否需要加購當地的旅遊行程#流程並未將旅遊行程導覽列入入住前七日的確認作業說明中。
~對飯店環境的滿意度預期#這屬於服務心理建設，而非流程中要求的實務作業確認項目。
}

::OP-FO-18-Q06::根據作業手冊，若發現顧客「匯款或預先授權」失敗，應在什麼時間點聯繫詢問顧客？ {
~發現失敗後立即取消訂單#流程應是先聯繫確認，而非直接執行取消動作。
=三日內尚未匯款或授權失敗時#手冊規範針對匯款或授權異常，需於三日內聯繫顧客處理。
~入住當天早上才進行聯繫#當天才聯繫過於倉促，會導致房源調度與收費風險增加。
~於入住前七日聯繫時一併詢問即可#支付異常屬於較緊急事項，流程規定應在發現後的「三日內」處理，而非併入七日聯繫。
~僅在系統註記，不需主動聯繫#流程明確要求需「聯繫詢問顧客」，以確保訂單的有效性。
}

::OP-FO-18-Q07::關於「入住前一日」的提醒作業，若顧客訂單中沒有提供電話號碼，應如何處置？ {
~直接省略提醒步驟，視同放棄#提醒作業是服務品質的一部分，不能因無電話就完全不處理。
=在 OTA 後台留言發送提醒資訊#流程規定入住前一日發簡訊，若無電話則在後台留言。
~嘗試在網路上搜尋顧客的其他聯繫方式#此舉耗時且可能涉及隱私問題，應使用官方規範的後台留言管道。
~改寄送紙本明信片至顧客住址#時效性不符，且流程中並無此項作業說明。
~聯絡該平台的區域業務窗口代為發簡訊#內部後台系統即具備留言功能，不需透過外部業務窗口操作。
}

::OP-FO-18-Q08::若旅客有特殊事項需要直接聯繫，但顧客未留電話，且飯店人員無法透過現有管道溝通時，最後的處理方式為何？ {
~自行取消該訂單以避開後續爭議#流程規定應透過客服協助，而非擅自取消訂單。
~在顧客抵達飯店時才當面溝通特殊事項#若為特殊緊急事項，需預先解決，以免現場無法處理導致客訴。
=致電平台客服人員，請客服人員代為聯繫顧客#當飯店方無法直接聯繫且有特殊需求時，請平台客服介入是標準 SOP。
~將訊息留在飯店大廳的告示板上#此方式缺乏針對性與隱私性，並非標準的溝通流程。
~聯繫顧客在預約單上填寫的緊急聯絡人#流程中並未指示聯繫緊急聯絡人，而是應透過平台客服代為聯繫。
}

::OP-FO-18-Q09::在飯店前檯系統執行「訂房確認」作業時，第一步與第二步的正確操作路徑為何？ {
~點選結帳管理 -> 點選發票查詢 -> 輸入訂單號碼#訂房確認應在訂房管理模組，而非結帳管理模組。
~點選客房狀態 -> 點選右方排房管理 -> 查看入住清單#雖然與入住相關，但標準的訂單確認流程是從「訂房管理」進入。
=點選訂房管理 -> 點選右方訂房管理 -> 輸入條件查詢並打開訂房單#這是手冊中定義的標準系統操作路徑，確保能正確檢視備註與訂單細節。
~點選報表管理 -> 點選每日入住表 -> 查詢特殊備註#報表管理用於統計與產出清單，非執行個別訂單確認與內容修改的路徑。
~點選會員管理 -> 點選歷史紀錄 -> 查詢該筆交易#會員管理無法處理即時的 OTA 訂房單狀態確認。
}

::OP-FO-18-Q10::在打開訂房單進行確認時，除了核對基本資訊，流程特別要求必須確認哪一個欄位內容？ {
~確認顧客的會員點數餘額#流程未提及需核對點數，重點在於訂單內容與支付狀態。
=確認備註內容，並看匯款、預先授權是否成功#手冊規範打開訂房單後需確認備註，並查核支付（匯款或授權）的狀態。
~確認該房型在其他平台的即時房價#房價比對是通路管理的範疇，非單筆訂房確認的標準作業細節。
~核對顧客過去在其他飯店的違規紀錄#飯店前檯系統通常不具備跨飯店的個人違規紀錄核對功能。
~確認顧客是否已下載飯店專屬 APP#此項目與訂房確認的核心作業流程無關。
}
