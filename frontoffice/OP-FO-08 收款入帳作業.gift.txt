$CATEGORY: OP-FO-08 收款入帳作業

::OP-FO-08-Q01::當一位顧客撥打電話告知已完成匯款，並提供帳號末五碼供核對時，櫃檯人員應登入玉山銀行網路銀行，並選擇哪一個帳號進行查帳作業？ {
~第一個帳號#根據匯款查帳作業標準，應固定點選第二個帳號進行明細查詢，而非第一個帳號。
=末五碼為 26358 的帳號#作業規範要求點選第二個帳號，其末五碼應為 26358，以確保核對的是正確的收款帳戶。
~與顧客匯款帳號相同的末五碼帳號#銀行帳號末五碼是指本館的收款帳號（26358），並非指要找一個與顧客帳號相同的收款帳號。
~隨機點選有交易紀錄的帳號#查帳需在特定的公司帳戶內進行，不可隨機選擇，否則會導致帳務核對錯誤。
}

::OP-FO-08-Q02::在執行信⽤卡預先授權時，若顧客提供的是「中國信託簽帳金融卡（卡號開頭為 4477-5787）」，櫃檯人員應如何反應最適當？ {
~照常進行預先授權操作#SOP 特別註記特定簽帳金融卡無法進行授權，強行操作將導致授權失敗。
=告知顧客此卡種無法授權，建議更換信用卡或改用匯款方式#針對無法授權的簽帳金融卡，應引導顧客提供其他信用卡或轉為匯款流程，以完成訂房保障。
~直接將訂單標示為無效並取消#應先與顧客溝通更換支付方式，不應在未通知的情況下直接取消訂單。
~請顧客聯繫銀行開啟預先授權功能#這是特定卡種的系統限制，而非開通功能即可解決，應優先請顧客更換支付工具。
}

::OP-FO-08-Q03::針對 Booking.com 的訂單，若在執行信用卡預先授權後不幸失敗，除了在 Booking.com 後台標示無效與傳送簡訊外，還必須在哪個系統中記錄以便後續追蹤？ {
=德安系統的備註欄位與 NOTION 的授權失敗表單#多處記錄（德安、Booking.com、NOTION）能確保內部團隊清楚掌握授權狀態，防止追蹤遺漏。
~僅在德安系統的客務帳包記錄#除了德安系統，SOP 要求需填入 NOTION 表單，以便管理端進行整體追蹤。
~玉山銀行網路銀行備註#授權失敗不會在銀行端產生成功入帳紀錄，因此無法也不需在銀行端備註。
~三竹簡訊系統的歷史紀錄#簡訊系統僅用於發送通知，並非正式的授權失敗追蹤與管理平台。
}

::OP-FO-08-Q04::顧客辦理入住時，需針對之前的預先授權做「補登交易（授權完成）」，此時刷卡機操作流程中，哪一組資訊是不需要輸入的？ {
~授權碼#補登交易必須對應原授權號碼，因此授權碼是必要資訊。
~系統追蹤號#追蹤號用於核對交易明細，是刷卡機完成補登流程的必填欄位。
=顧客的匯款帳號末五碼#匯款帳號資訊屬於「匯款流程」，在信用卡授權補登交易中完全不需要用到。
~序號#補登流程中需依照序號進行對應，以確保該筆授權被正確完成。
}

::OP-FO-08-Q05::在德安系統執行「入訂金流程」時，若一位顧客預計入住 5 天，且明確要求「最後只需要一張總發票」，櫃檯人員應如何選擇發票開立方式？ {
~選擇「後開」#「後開」通常用於每天都要產出一張發票的情況，不符合顧客要求單一總發票的需求。
~選擇「不開立」#無論顧客需求為何，入帳後皆須設定發票開立邏輯，不可略過此步驟。
=選擇「已開」#根據 SOP 補充說明，若入住多天僅需一張總發票，應選擇「已開」選項以符合帳務處理邏輯。
~選擇「待確認」#系統中並無此預設標準選項，應依據顧客需求直接選擇對應的發票處理方式。
}

::OP-FO-08-Q06::當顧客臨時更改付款方式，需取消之前的預先授權時，關於退還額度的時間點，櫃檯人員應如何向顧客說明？ {
~刷卡機點選取消後，額度會立即恢復#取消授權後銀行端仍需作業時間，並非即時恢復額度，不應給予顧客錯誤期待。
=申請約需 7 到 14 個工作天，依銀行作業而定#這是 SOP 規定提供給顧客的標準話術，能專業地說明銀行處理的預計週期。
~需等到顧客下次收到帳單時才會歸還#授權取消與實際請款後的刷退不同，額度恢復通常在 7-14 個工作天內，而非等到帳單結帳。
~退還時間由德安系統自動決定，無法預知#退還額度是銀行的作業範圍，與飯店的德安前台系統無關。
}

::OP-FO-08-Q07::在執行「結帳流程」時，若該筆訂單來源為訂房網站（如 Booking.com），且需開立發票給顧客，則付款方式代號應選擇哪一個？ {
~00 沖客房訂金#「00 沖客房訂金」僅適用於現場已有預收訂金款項的情境。
=客⼾簽帳#SOP 補充說明提到，除了有收款沖訂金外，其餘訂房網站應選擇「客戶簽帳」並輸入代號。
~住宿券#住宿券僅適用於顧客持券消費的情境，不適用於一般訂房網站結帳。
~現金結帳#訂房網站通常涉及佣金或線上支付，與一般的現金結帳流程有所區別。
}

::OP-FO-08-Q08::顧客在入住期間購買了飯店的備品或產生了額外消費（其他費用），櫃檯人員應如何在德安系統中進行「入帳」操作？ {
~進入訂房系統點選入訂金流程#額外消費屬於住客帳處理，不應與預收訂金流程混淆。
=在房間管理找到旅客後，點選「旅客帳處理」再點選「入帳」#這是針對在店旅客產生非房費消費時，標準的系統操作路徑。
~直接在住客帳維護中點選預估款#預估款用於結帳流程前的確認，新增單筆消費應從旅客帳處理進入。
~在公帳號設定中產生新的公帳號#公帳號用於特定帳務拆分或轉帳，而非記錄單一旅客的即時消費。
}
