$CATEGORY: OP-FO-22 發票異常作業處理

::OP-FO-22-Q01::櫃檯夥伴在執行早班交接作業時，若要確認前一日的發票是否有「跳開」的異常情形，應進入德安系統的哪個路徑進行核對？ {
~進入「發票管理」並點選「發票查詢」#發票管理主要用於單筆發票的處理與作廢，而非用於核對整批開立的序號連續性。
~點選「系統維護」並查看「連線日誌」#系統維護主要處理連線異常，無法直接看到發票號碼的開立明細與跳號情形。
=點選「報表」中的「發票開立明細表」#透過報表產出的明細表，能完整列出特定日期區間內的所有發票號碼，便於人工核對號碼是否連續。
~進入「後台開立發票」檢查剩餘張數#後台開立是用於補開發票，無法提供過去已開立發票的完整核對清單。
}

::OP-FO-22-Q02::在檢查報表時，若發現發票號碼確有「跳開」的情況，根據異常處理流程，下一步最正確的做法為何？ {
~直接手開一張三聯式發票給顧客#手開發票是備援機制，系統跳開應先透過系統流程處理作廢與重開，以確保帳務與發票庫存一致。
=將該跳開的發票號碼進行作廢，再由後台重開#這是標準的異常補救程序：先處理錯誤的（作廢），再補足正確的（後台重開）。
~在備註欄標記「系統跳號」並忽略不計#跳號會造成發票字軌不連續，若未進行作廢程序，會導致稅務申報出現異常。
~通知顧客退回原發票，再由前台重新過帳#跳開通常是指號碼遺漏，原發票若已交付顧客且金額正確則不需退回，僅需針對跳過的空號處理。
}

::OP-FO-22-Q03::在執行「作廢發票」流程時，於德安系統選取特定發票後，應如何執行作廢動作？ {
~直接按鍵盤上的「Delete」鍵#系統安全性設定不會允許直接透過刪除鍵作廢發票，以防誤觸。
=對該發票點選滑鼠右鍵選擇「作廢發票」#這是系統介面中指定的功能路徑，透過右鍵選單進行狀態變更。
~雙擊發票進入編輯頁面後刪除金額#發票開立後金額不可直接修改，必須執行完整的作廢流程。
~點選視窗上方的「檔案」選擇「取消交易」#取消交易與發票作廢在系統中是不同的作業邏輯。
}

::OP-FO-22-Q04::關於「已作廢發票」的實體文件處理，下列哪一項敘述符合標準作業流程？ {
~為了環保，應將作廢發票背面當作便條紙使用#發票為具名稅務憑證，作廢後仍須存查，不可隨意挪作他用。
=需蓋上作廢章並黏貼到指定的作廢發票表格#這是為了滿足財務審核與稅務查核的需求，確保每一張作廢發票都有跡可循且經過標記。
~直接投入碎紙機以防顧客資料外洩#實體發票必須保留以供核對，直接銷毀會造成帳實不符。
~將發票對折後收入抽屜，等月底再交給會計#SOP 要求需進行蓋章與黏貼動作，而非單純存放，以避免遺失或重複誤用。
}

::OP-FO-22-Q05::若顧客入住三天以上，在執行「後台開立發票」作業時，應該開立幾張發票？ {
~每天分別開立一張發票#對於同一筆住宿交易，為了作業簡化與符合內部規範，不需逐日拆分開立。
=只要開立一張發票即可#SOP 明確規定顧客入住一天以上發票只要一張，避免資源浪費與核帳困難。
~視顧客付款次數而定，付幾次就開幾張#後台補開通常是針對整筆入住需求處理，不應隨意拆分張數。
~必須開立一張住宿費與一張雜項費#除非顧客有特殊需求且符合規定，否則預設應以單張處理。
}

::OP-FO-22-Q06::櫃檯夥伴在二館執行後台補開發票時，在「銷售點代號」欄位應選取哪一個選項？ {
~財務管理中心#這是行政部門代碼，非現場銷售點，選取後會導致營收歸戶錯誤。
=客房櫃檯(二館櫃台)#選取正確的銷售點代號，能確保帳務準確歸類至該館別與服務點。
~一館接待總機#雖然同屬旅館，但跨館別選取會造成報表統計與稅務申報的混亂。
~一般銷售(通用)#應優先選擇特定的銷售點代號，以利後續帳務追蹤。
}

::OP-FO-22-Q07::進行「手開三聯流程」時，若房價總計為 $3150$ 元，請問「銷售額」欄位應填寫多少金額？ {
~$3150$ 元#$3150$ 是含稅總價，銷售額應為不含稅的淨額。
=$3000$ 元#計算方式為總價除以 $1.05$（$3150 \\div 1.05 \= 3000$），這才是正確的銷售額。
~$150$ 元#$150$ 元是該金額的營業稅（$3000 \\times 0.05$），而非銷售額。
~$3307$ 元#這是將總價再次加稅的錯誤計算方式。
}

::OP-FO-22-Q08::填寫手開三聯式發票時，規範中特別提醒要「注意日期是否回開」，其主要目的是為了預防什麼問題？ {
~預防顧客不滿意入住日期#發票日期與滿意度無直接關係，主要涉及法律與會計遵循。
=確保發票開立時序符合稅務法令規範#發票日期若晚於目前的發票序號區間（回開），會違反稅務機關對字軌序號與日期先後的規定。
~避免電腦系統無法過帳#手開發票不直接連動系統過帳，主要是實體文件的合法性問題。
~防止顧客報帳時日期錯誤#雖然也是原因之一，但更核心的是旅館端的申報合法性風險。
}

::OP-FO-22-Q09::完成手開三聯式發票後，關於各聯次的交付與留存，下列做法何者正確？ {
~第一聯交付顧客，其餘留存#第一聯為存根聯，應留在旅館存查。
=第二、三聯交付顧客，其餘留存#第二聯為扣抵聯，第三聯為收執聯，皆需交付顧客作為報帳或存證之用。
~將三聯全部交給顧客，避免爭議#旅館若無留存第一聯存根，將無法進行後續帳務核對與申報。
~僅交付第三聯給顧客即可#若顧客有開立統編需求（三聯式），通常需要扣抵聯（第二聯）才能進行稅務抵減。
}

::OP-FO-22-Q10::在執行完發票作廢流程後，SOP 規定必須「離開後重複以上動作複查」，這項操作的目的是？ {
~確認顧客是否已經離開現場#這與系統內的發票作廢確認無關。
=確保系統確實執行成功並顯示已作廢狀態#為了預防系統偶發性的儲存失敗或人為漏看，重複檢查可確保作廢狀態已生效。
~為了練習系統操作熟練度#SOP 的目的是確保資料正確性，而非單純的教學練習。
~重新確認發票上的金額是否正確#既然已經要作廢，金額正確與否已非重點，重點在於作廢狀態的確立。
}
