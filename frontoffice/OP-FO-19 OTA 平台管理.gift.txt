$CATEGORY: OP-FO-19 OTA 平台管理

::OP-FO-19-Q01::在每日開始作業時，針對 OTA 平台的「房量與價格巡檢」工作，夥伴應優先執行的關鍵動作為何？ {
~確認飯店官網的房量是否與所有 OTA 平台同步。#雖然官網同步很重要，但 SOP 強調的是每日需對比 PMS 系統與 OTA 後台的庫存數據一致性。
=比對內部 PMS 系統與各個 OTA 平台上的庫存數量是否一致。#確保飯店管理系統（PMS）與對外販售平台（OTA）數據同步，是預防超賣與超售的基礎步驟。
~直接調整當日房價以吸引最後一刻的訂單。#巡檢的目的在於確認資訊正確性，盲目調價而不先確認庫存與專案內容可能導致後續售價混亂。
~只針對住房率超過百分之九十的日期進行抽查。#SOP 規定應「每日」確認主要平台狀態，不限於高住房率日期，儘管高住房日需增加頻率。
}

::OP-FO-19-Q02::當飯店因應重要節日或競品狀況需要調整「浮動房價」時，執行過程中哪一個環節最能避免設定錯誤？ {
=調整完畢後，必須再次進入平台前端或後台檢查是否正確生效。#設定後的二次檢查能確保操作無誤，避免因系統延遲或設定漏失導致售價異常。
~憑藉過往經驗，直接在多個平台同時進行大範圍調價。#調價應依據主管指示與市場狀況，且在大範圍更動前應先確認適用日期與房型，而非僅憑經驗。
~為了搶占市場先機，調整後立即關閉調價紀錄表以節省時間。#SOP 明確要求調價需有紀錄，不可任意變動，紀錄是未來異常追蹤的重要依據。
~先調整 PMS 價格，OTA 端的價格等有客人反應再修正。#調價需確認適用的平台範圍並確保生效，被動等待客人反映會造成現場服務與售價爭議。
}

::OP-FO-19-Q03::在預見到未來某個熱門活動日可能出現「超賣風險」時，現場夥伴應採取何種預防措施？ {
~等到當天有訂單重疊時，再聯繫客人取消訂單。#這屬於事後應變而非預防，會造成嚴重的客訴與平台罰則。
=依指示針對房況緊張的日期調整停售、關房或下修可售數。#在風險發生前主動控制庫存（下修可售數或關房），是防止系統自動超賣的標準預防動作。
~將該日期所有房型價格調至最高，以嚇阻客人訂房。#單純調高價格無法保證不會成交，直接管理可售數量才是更有效且專業的預防方式。
~暫時關閉 PMS 系統，只保留 OTA 平台的運作。#關閉 PMS 會導致現場排房混亂，正確的做法是調整 OTA 平台上的可售房數。
}

::OP-FO-19-Q04::若不幸發生超賣（Overbooking）且已確認受影響的訂單，第一時間的正確處理流程為何？ {
~先致電客人表示房型設備維修，誘導客人主動取消。#對客說明語氣應穩定且誠實，推託或欺騙會損害飯店聲譽且違反 SOP 注意事項。
=立即整理受影響訂單資訊並回報主管，依指示進行移房或協調。#超賣案件需主管介入，且必須先整理完整的訂單與付款資訊，才能進行有效的決策與轉單處理。
~自行聯繫鄰近飯店轉單，處理完畢後再向主管口頭報備。#SOP 規定超賣需「立即回報主管」並「依指示」處理，不應由夥伴在未報告前自行決定轉單。
~在 OTA 後台直接按拒絕訂單，不需保留任何紀錄。#平台端需同步保留處理紀錄，且直接拒絕訂單可能導致平台高額罰款。
}

::OP-FO-19-Q05::關於 OTA 平台訊息與訂單追蹤的日常作業，下列哪一種做法符合 SOP 的要求？ {
=若發現客人信用卡無效，應立即透過平台訊息留言或請客服協助。#針對無效卡或特殊需求，依平台規則聯繫是標準流程，能確保訂單的有效性。
~為了展現熱忱，應直接取得客人私人的通訊軟體聯繫方式進行溝通。#SOP 指示應依平台規則留言或轉由客服協助，以確保溝通紀錄受平台保障。
~平台訊息只需在退房當天確認有無特殊備註即可。#訊息應於規定時限內處理，延遲處理可能導致客人取消或需求無法被滿足。
~若客服通知有訂單變更，口頭記下後等有空再處理。#平台訊息與通知應及時追蹤，且異常事項必須有紀錄，不能僅靠口頭記憶。
}

::OP-FO-19-Q06::當班次結束前，若當天曾發生「平台系統異常」且尚未完全修復，夥伴應如何處理紀錄與交接？ {
=在交接班或主管追蹤表上清楚記錄，並確認接手人已充分了解現況。#未結案的異常案件必須清楚說明，並記錄於指定表格，以確保跨班次處理的連續性。
~因為尚未結案，所以不需記錄在追蹤表，等明天處理完再寫。#異常紀錄應即時記錄於交班表，延後紀錄會導致資訊斷層。
~口頭告訴下一班夥伴要注意，不需額外書寫文件以免留下負面紀錄。#SOP 要求必須將異常紀錄於交班或主管追蹤表，單純口頭交接極易造成遺漏。
~為了準時下班，先在追蹤表上註記「已結案」，讓下一班重新啟案即可。#虛假結案會導致異常案件無法被追蹤，違反 SOP 對異常紀錄與結案的基本要求。
}
