$CATEGORY: OP-FO-07 入住接待標準作業指導書

::OP-FO-07-Q01::當客人在中午 13\:30 抵達飯店且房間已經清潔完畢時，根據標準作業程序，櫃檯人員應該如何處理？ {
~直接為客人辦理免費提前入住#免費提前入住的標準時段通常設定在 14\:00 之後，在此之前需視情況判斷。
~請客人在大廳等到 15\:00 正式入住時間#這忽略了房間已清潔完畢的彈性處理空間，可能會降低顧客滿意度。
=詢問主管是否需要向客人收取提前入住費用#在 12\:00 至 14\:00 之間，若房間已好且顧客有需求，應先請示主管關於收費的決策。
~告知客人現在入住需加收每小時兩百元的加休費#加休費通常適用於 15\:00 之前的逆推計算，但此情境下應先確認主管是否因特殊狀況免收或另行規範。
}

::OP-FO-07-Q02::若在系統中查不到顧客訂單，但顧客確認有預訂且是透過 WEB 訂購，第一步應如何操作？ {
~立即為顧客手建一筆新單#在手建新單前，應先嘗試系統同步，以避免造成訂單重複或來源紀錄錯誤。
=確認是否為系統未下傳訂單，並嘗試手動執行下傳#優先確認系統對接狀態並手動同步，能確保訂單資料與來源平台一致。
~請顧客聯繫原訂房平台處理，飯店不予受理#這違反了應先處理顧客當下需求、後續再處理訂單問題的服務標準。
~要求顧客提供訂單截圖後直接入住，不需入系統#所有入住皆須在系統內有正式紀錄，以免影響後續帳款與房況控管。
}

::OP-FO-07-Q03::關於「大和停車場」的補助規範，下列哪一項敘述是正確的？ {
~補助時間可達 24 小時，且不限次數進出#大和停車場的補助有明確的截止時間點，而非固定的時數區間。
~離場時不需要告知櫃檯，系統會自動辨識車號#該停車場需透過人工操作電腦消磁，因此顧客必須在離場前告知櫃檯。
=補助僅提供至退房當天的中午 12 點，逾時需自付#這是該特定配合停車場的系統設定限制，超過中午 12 點後電腦會自動消磁失效。
~顧客進出停車場時，每次都需要輸入不同的隨機密碼#規範要求進出必須輸入相同的四位數字，以確保系統辨識的一致性。
}

::OP-FO-07-Q04::當特約停車場全部滿車，且顧客表示入住後「不會再移車」時，應指引顧客停往何處？ {
~請顧客自行尋找路邊停車格，飯店不予補助#對於不再移車的顧客，飯店有備用的補償方案以維持服務品質。
=引導至民權停車場，退房時憑發票打統編請款#針對不移車的顧客，這是特約全滿時的標準應變流程，透過核銷方式補助客人。
~請顧客停在飯店門口紅線，由櫃檯代為看管#這違反交通法規且存在違規受罰風險，並非合規的作業程序。
~建議顧客改停至大和 TIMES 1，因為該場地永不客滿#所有特約場地都有滿位的可能，且 SOP 針對全滿狀況有特定指引。
}

::OP-FO-07-Q05::在辦理入住登記（R卡）時，若對象為外國顧客，除了基本資料外還必須特別留下哪項資訊？ {
~家長同意書#這是針對未成年者的要求，而非所有外國顧客。
~社群媒體帳號 (如 Line 或 WeChat)#這並非官方登記要求的標準必填項目。
~在台聯繫人的電話號碼#雖然有用，但 SOP 規定的標準必填項是聯繫用的電子通訊方式。
=電子郵件 (E-MAIL)#針對外國顧客，Email 是確保能後續聯繫與傳送帳單資訊的重要管道。
}

::OP-FO-07-Q06::若顧客先前已完成信用卡預先授權，但入住現場想改用「現金」支付，櫃檯應如何處理？ {
~告知顧客預先授權無法取消，僅能使用原卡支付#顧客有權利更改付款方式，櫃檯應協助進行額度釋放。
~直接收取現金，不需要對原信用卡做任何處置#若不取消授權，會導致顧客信用額度被持續凍結，引發爭議。
=申請取消預先授權將額度歸還，再收取現金房費#正確流程是釋放先前凍結的額度，再依照顧客新的支付需求進行入帳。
~先以現金結帳，待退房後系統會自動解除預先授權#預先授權需由飯店主動申請取消，否則銀行端釋放時間較慢，可能造成客人困擾。
}

::OP-FO-07-Q07::在德安系統入訂金流程中，若顧客是續住且希望「全部開成一張發票」，發票開立方式應選取？ {
~後開#「後開」通常用於單日開立或一天一張發票的情況。
=已開#針對續住或需要合併開立的情境，系統邏輯需設定為「已開」以符合統整開立的需求。
~暫緩#系統選項中並無「暫緩」作為標準發票開立狀態的分類。
~免開#除了特定免稅實體，一般住房皆須依法開立發票。
}

::OP-FO-07-Q08::若住客反應忘記帶房卡且目前房內無人，櫃檯人員查驗身分無誤後應如何處理？ {
~直接在櫃檯重發一張新卡交給住客#基於安全考量，應由員工陪同確認或帶領入內，而非直接交付感應卡。
~請住客在大廳等待，直到房務人員巡樓時順便幫忙開門#這會造成顧客不便且缺乏效率，應由櫃檯主動協助。
=親自帶領客人上樓並協助開啟房門#這是確認身分後最安全的標準協助程序，能避免卡片外流風險。
~要求住客支付房卡遺失工本費後才提供協助#「忘記帶卡」不代表「卡片遺失」，且首要任務是協助入內而非收費。
}

::OP-FO-07-Q09::在德安系統中執行 CHECK IN 步驟時，若遇到一筆訂單有多間房間，應注意什麼？ {
~系統會自動感應所有房間同時入住，不需手動操作#系統仍需人工勾選與確認，無法完全自動感應多房狀態。
=確認是否所有房間都要同時辦理入住，避免遺漏#多房訂單可能分批抵達，操作時須確認勾選範圍以確保每間房都正確入帳。
~必須分次建立不同的訂單編號才能處理#同一筆訂單即可管理多房，分開建單會增加作業複雜度。
~多間房間的訂單只能由主管帳號執行入住#這是前檯人員的標準作業範疇，不需特定權限。
}

::OP-FO-07-Q10::關於信用卡簽單的操作，即便系統顯示為「免簽名」，櫃檯人員仍應如何要求顧客？ {
~告知顧客免簽名是銀行的規定，直接讓顧客離開#雖然銀行端免簽，但飯店會計與內控流程仍有簽名核對的需求。
=請顧客在簽單空白處簽名，以供會計核對身分#這是為了內部稽核與未來潛在爭議核對身分的重要步驟。
~要求顧客提供身分證件拍照留存，代替簽名#隨意拍攝證件涉及資安風險，簽名是較合適的輔助確認方式。
~向顧客說明系統故障，必須強制簽名才能完成交易#不應以錯誤資訊誤導顧客，應誠實說明是為了內部核對身分之用。
}
